# Professional Strategy Backtest Audit Engine v2

## 1. 定位

Professional Strategy Backtest Audit Engine v2 是川 Market Radar 的核心能力体检系统。

它不是量化收益回测，不自动下单，不证明历史收益，不自动修改策略权重。它的唯一目标是用历史行情全面审计网站当前的扫描、分析、推理、风控、交易计划和复盘进化能力。

旧版 `npm run backtest:historical` 只保留为 smoke test：它能验证历史 K 线回放和早期雷达评分是否能跑通，但不能代表完整网站分析能力。

## 2. 硬边界

- 只读历史数据，不写数据库。
- 不自动下单，不连接交易所下单 API。
- 不自动调整实时权重。
- 不把回测结果包装成收益承诺。
- 不允许偷看未来数据。
- 不允许用 mock、0 值、当前值或旧缓存冒充历史数据。
- 数据缺失必须标记 `unavailable` 或 `partial`。
- 公开交易所历史 Funding/Open Interest 可以作为 `public_exchange` 审计证据，但不能冒充 CoinGlass。
- 历史衍生品取样必须只使用观测时间之前的数据，严禁未来函数。
- 每轮回测必须输出发现的问题、问题归因和整改方案。

## 3. 被审计能力

完整测试方案见 `docs/backtest-v2/BACKTEST_TEST_PLAN.md`。本规格定义系统边界和必须审计的能力，测试方案定义每轮具体要测什么、怎么抽样、怎么看问题和怎么判定有效。

### 3.1 全市场扫描能力

检查历史时间点上，真正启动或大幅下跌的山寨币是否被系统发现。

必须审计：

- 是否覆盖全市场 USDT 永续池。
- 是否只盯固定大币。
- 候选池是否被固定币霸占。
- 新币、小币、异动币是否有机会进入深扫。
- 每个币为何进入或没进入下一层。

### 3.2 提前发现能力

每个信号必须标记发现位置：

- `PRE_MOVE`：行情启动前。
- `EARLY_MOVE`：刚启动。
- `MID_MOVE`：已经启动一段。
- `LATE_MOVE`：已经涨完或跌完。
- `AFTER_MOVE`：事后信号。

偏晚信号不能算成功。

### 3.3 候选筛选能力

检查 Top5 / Top10 / Top20 是否优于：

- 随机选币。
- 24h 涨幅榜。
- 24h 跌幅榜。
- 成交额榜。
- 单纯成交量异动。
- 单指标策略。

如果完整系统跑不赢简单基线，报告必须明确输出阻断问题。

### 3.4 结构分析能力

必须审计：

- 前高前低。
- 箱体。
- 支撑压力。
- 波动压缩。
- 吸筹。
- 突破前。
- 突破确认。
- 假突破。
- 回踩质量。
- 趋势延续。
- 衰竭风险。

结构判断必须优先于技术指标。

### 3.5 技术指标解释能力

必须审计：

- RSI。
- MACD。
- Bollinger。
- ATR。
- EMA。
- VWAP。
- ADX。
- OBV / CVD 近似。
- Fibonacci。
- 成交量均线。
- 波动率压缩。

指标只做辅助。任何单一指标不得直接生成交易结论。

### 3.6 多周期推理能力

必须审计：

- 15m / 1h / 4h / 1d 是否一致。
- 低周期信号是否错误推翻高周期。
- 高周期压力是否正确拦截短周期多头。
- 高周期支撑是否正确拦截短周期空头。
- 日线或周线双熊时，多头是否被降级。
- 市场环境窗口必须分清楚：短周期 `4-24h`，中周期 `3-7 天`，长周期 `30-90 天`，大级别趋势背景 `1d + 1w`。`--days 30` 只是默认历史取数窗口和长周期下限，不等于唯一长周期判断。
- 专业审计节点必须使用分层验证窗口：small 节点默认看未来 4h，medium 节点默认看未来 24h，large 节点默认看未来 96h。报告和前端进度必须保留每个节点的 `validationWindowLabel`，不能把所有周期都包装成同一个 24h 结论。

### 3.7 衍生品验证能力

必须审计：

- CoinGlass OI。
- CoinGlass Funding。
- CoinGlass 多空拥挤。
- Binance / OKX / Bybit 公开 Funding。
- Binance / OKX / Bybit 公开 OI。
- 主动买卖和 CVD 近似。

如果 CoinGlass 某端点受套餐限制，报告必须写清楚，不允许用公开交易所数据冒充 CoinGlass。

### 3.8 风险赔率能力

每个计划必须审计：

- 入场触发。
- 结构止损。
- TP1 / TP2 / TP3。
- 实际 RR。
- 先到 TP 还是先到 SL。
- 是否超时。
- 是否追涨追跌。

RR 本体按结构距离计算。用户个人杠杆和 0.3% 保证金只用于个人风险展示，不改变结构 RR。

### 3.9 交易计划能力

计划必须能回答：

- 为什么看。
- 怎么进。
- 哪里错。
- 哪里止损。
- 哪里止盈。
- 什么情况失效。

只有 `TRADE_PLAN_READY` 可进入交易计划审计；其他状态只做候选或证据审计。

### 3.10 复盘进化能力

必须审计：

- 哪类信号有效。
- 哪类信号坑人。
- 哪些机会漏掉。
- 哪些规则误伤。
- 哪些规则放过垃圾机会。
- 下一轮应该增强或降权什么。

## 4. 专业回测流程

每个历史回放点必须按真实网站链路执行：

1. 读取当时以前的数据。
2. 重建当时的全市场轻扫池。
3. 生成候选池。
4. 执行深扫验证。
5. 生成证据链。
6. 执行技术指标解释。
7. 执行结构和多周期判断。
8. 执行风险门控和 RR 判断。
9. 生成或拒绝交易计划。
10. 追踪未来结果。
11. 输出复盘归因。

### 4.1 10x10 专业审计轮次

正式审计轮次必须支持：

- 10 个不同类型山寨币目标样本。
- 每个币 10 个不同历史节点。
- 节点必须覆盖小周期、中周期、大周期。
- 节点类型必须至少覆盖：启动前、早期放量、突破边缘、回踩、趋势加速、晚到延展、假突破/失效、随机中性、中周期波段、大周期背景。
- 每个节点必须输出：是否被 radar topN 捕获、雷达排名、是否迟到、最大浮盈、最大回撤、命中、成熟度、问题数量。
- 前端 `/review` 必须能读取 `latest-progress.json`，展示运行中、失败或已完成状态，不能等最终报告生成后才有反馈。

### 4.2 目标池和候选池必须分离

10 个目标币只是审计样本，不等于候选排序池。

硬规则：

- 专业审计必须采用“目标币池 + 候选大池”模式。
- 默认目标币：10 个。
- 默认候选池：至少 80 个。
- 每轮 TopN 必须显著小于候选池，默认 Top10。
- 禁止用 10 个币选 Top10 来证明扫描有效。
- 如果候选池数量不大于 TopN，报告必须输出 `PBA-SCAN-ROUND-DESIGN-001`，并判定该轮扫描捕获率和基线对比无效。
- 报告必须同时展示目标币数量、候选池数量和 TopN。

原因：如果候选池和目标币数量一样，系统天然“捕获”所有目标，不能证明它有全市场筛选能力。

## 5. 每轮报告必须包含

- 本轮参数。
- 数据源可用性。
- 使用币种数。
- 回放时间点数。
- 交易所覆盖。
- 雷达与基线对比：`radar` / `momentum` / `volume` / `random` 都必须输出样本数、命中率、迟到率、平均 MFE、平均 MAE、入选时已波动幅度和成交量倍数。
- 10x10 审计进度：目标币、候选池、完成节点、捕获率、当前节点、最近节点样本。
- 提前性审计：必须输出提前样本、迟到样本、迟到率和无交易计划样本数；偏晚信号不能被包装成成功。
- 技术指标有效性。
- 结构判断有效性。
- 多周期门控有效性。
- 衍生品证据有效性。
- RR 和交易计划有效性。
- 典型成功案例。
- 典型失败案例。
- 漏判机会：必须列出未进入 radar topN 但事后达到阈值的可学习样本，并说明只能用于复盘校准，不能自动修改实时权重。
- 漏判归因：必须输出目标在 radar 排序中的名次、主要漏判节点类型、主要漏判币种类型和平均排序差距；只说“漏掉了”不合格。
- radar 排名不能只使用单一信心分。专业审计必须把不晚到的回踩/反抽再确认、压缩、早期放量作为提前机会加分项，把已经大幅涨跌、位置极端、高波动追涨样本作为降权项。
- 迟到信号。
- 计划就绪必须经过最后质量防线：结构止损、目标方向、RR、止损距离全部通过后才允许进入 `TRADE_PLAN_READY`；上游模块误标可交易时，计划层必须能再次阻断。
- 每个审计节点必须写明验证窗口，避免把短线机会、中线结构和大周期背景混成一个结论。
- 每轮报告必须写明市场环境窗口，避免把 30 天默认取数窗口误读成唯一长周期市场环境。
- 问题清单。
- 问题归因。
- 下一轮整改清单。

## 6. 问题编号

- `PBA-DATA-*`：历史数据或数据可用性。
- `PBA-SCAN-*`：全市场扫描和候选池。
- `PBA-TIMING-*`：提前性和迟到信号。
- `PBA-STRUCTURE-*`：结构判断。
- `PBA-INDICATOR-*`：技术指标解释。
- `PBA-TIMEFRAME-*`：多周期推理。
- `PBA-DERIVATIVES-*`：衍生品验证。
- `PBA-RR-*`：风险赔率。
- `PBA-PLAN-*`：交易计划。
- `PBA-REVIEW-*`：复盘进化。

## 7. 验收标准

专业回测 v2 完成后，必须能回答：

1. 网站是否真的提前发现了机会。
2. 网站是否跑赢简单基线。
3. 网站哪些分析规则有效。
4. 网站哪些分析规则误导。
5. 网站为什么漏掉机会。
6. 网站为什么放过垃圾信号。
7. 下一轮必须修什么。

如果不能回答这些问题，不能称为专业回测系统。

一轮正式回测还必须满足 `docs/backtest-v2/BACKTEST_TEST_PLAN.md` 的有效性标准：候选池数量显著大于 TopN、完成 10 类目标币和 100 个历史节点、输出四条基线、输出提前性和漏判机会、输出问题编号/根因/整改方案，并且 `/review` 能看到最新进度或最新报告。
