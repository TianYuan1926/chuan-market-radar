# Professional Strategy Backtest Audit Engine v2

## 1. 定位

Professional Strategy Backtest Audit Engine v2 是川 Market Radar 的核心能力体检系统。

它不是量化收益回测，不自动下单，不证明历史收益，不自动修改策略权重。它的唯一目标是用历史行情全面审计网站当前的扫描、分析、推理、风控、交易计划和复盘进化能力。

旧版 `npm run backtest:historical` 只保留为 smoke test：它能验证历史 K 线回放和早期雷达评分是否能跑通，但不能代表完整网站分析能力。

## 2. 硬边界

- 只读历史数据，不写数据库。
- 不自动下单，不连接交易所下单 API。
- 不自动调整实时权重。
- 不把回测结果包装成收益承诺。
- 不允许偷看未来数据。
- 不允许用 mock、0 值、当前值或旧缓存冒充历史数据。
- 数据缺失必须标记 `unavailable` 或 `partial`。
- 公开交易所历史 Funding/Open Interest 可以作为 `public_exchange` 审计证据，但不能冒充 CoinGlass。
- 历史衍生品取样必须只使用观测时间之前的数据，严禁未来函数。
- 每轮回测必须输出发现的问题、问题归因和整改方案。

## 3. 被审计能力

### 3.1 全市场扫描能力

检查历史时间点上，真正启动或大幅下跌的山寨币是否被系统发现。

必须审计：

- 是否覆盖全市场 USDT 永续池。
- 是否只盯固定大币。
- 候选池是否被固定币霸占。
- 新币、小币、异动币是否有机会进入深扫。
- 每个币为何进入或没进入下一层。

### 3.2 提前发现能力

每个信号必须标记发现位置：

- `PRE_MOVE`：行情启动前。
- `EARLY_MOVE`：刚启动。
- `MID_MOVE`：已经启动一段。
- `LATE_MOVE`：已经涨完或跌完。
- `AFTER_MOVE`：事后信号。

偏晚信号不能算成功。

### 3.3 候选筛选能力

检查 Top5 / Top10 / Top20 是否优于：

- 随机选币。
- 24h 涨幅榜。
- 24h 跌幅榜。
- 成交额榜。
- 单纯成交量异动。
- 单指标策略。

如果完整系统跑不赢简单基线，报告必须明确输出阻断问题。

### 3.4 结构分析能力

必须审计：

- 前高前低。
- 箱体。
- 支撑压力。
- 波动压缩。
- 吸筹。
- 突破前。
- 突破确认。
- 假突破。
- 回踩质量。
- 趋势延续。
- 衰竭风险。

结构判断必须优先于技术指标。

### 3.5 技术指标解释能力

必须审计：

- RSI。
- MACD。
- Bollinger。
- ATR。
- EMA。
- VWAP。
- ADX。
- OBV / CVD 近似。
- Fibonacci。
- 成交量均线。
- 波动率压缩。

指标只做辅助。任何单一指标不得直接生成交易结论。

### 3.6 多周期推理能力

必须审计：

- 15m / 1h / 4h / 1d 是否一致。
- 低周期信号是否错误推翻高周期。
- 高周期压力是否正确拦截短周期多头。
- 高周期支撑是否正确拦截短周期空头。
- 日线或周线双熊时，多头是否被降级。

### 3.7 衍生品验证能力

必须审计：

- CoinGlass OI。
- CoinGlass Funding。
- CoinGlass 多空拥挤。
- Binance / OKX / Bybit 公开 Funding。
- Binance / OKX / Bybit 公开 OI。
- 主动买卖和 CVD 近似。

如果 CoinGlass 某端点受套餐限制，报告必须写清楚，不允许用公开交易所数据冒充 CoinGlass。

### 3.8 风险赔率能力

每个计划必须审计：

- 入场触发。
- 结构止损。
- TP1 / TP2 / TP3。
- 实际 RR。
- 先到 TP 还是先到 SL。
- 是否超时。
- 是否追涨追跌。

RR 本体按结构距离计算。用户个人杠杆和 0.3% 保证金只用于个人风险展示，不改变结构 RR。

### 3.9 交易计划能力

计划必须能回答：

- 为什么看。
- 怎么进。
- 哪里错。
- 哪里止损。
- 哪里止盈。
- 什么情况失效。

只有 `TRADE_PLAN_READY` 可进入交易计划审计；其他状态只做候选或证据审计。

### 3.10 复盘进化能力

必须审计：

- 哪类信号有效。
- 哪类信号坑人。
- 哪些机会漏掉。
- 哪些规则误伤。
- 哪些规则放过垃圾机会。
- 下一轮应该增强或降权什么。

## 4. 专业回测流程

每个历史回放点必须按真实网站链路执行：

1. 读取当时以前的数据。
2. 重建当时的全市场轻扫池。
3. 生成候选池。
4. 执行深扫验证。
5. 生成证据链。
6. 执行技术指标解释。
7. 执行结构和多周期判断。
8. 执行风险门控和 RR 判断。
9. 生成或拒绝交易计划。
10. 追踪未来结果。
11. 输出复盘归因。

## 5. 每轮报告必须包含

- 本轮参数。
- 数据源可用性。
- 使用币种数。
- 回放时间点数。
- 交易所覆盖。
- 雷达与基线对比。
- 技术指标有效性。
- 结构判断有效性。
- 多周期门控有效性。
- 衍生品证据有效性。
- RR 和交易计划有效性。
- 典型成功案例。
- 典型失败案例。
- 漏判机会。
- 迟到信号。
- 问题清单。
- 问题归因。
- 下一轮整改清单。

## 6. 问题编号

- `PBA-DATA-*`：历史数据或数据可用性。
- `PBA-SCAN-*`：全市场扫描和候选池。
- `PBA-TIMING-*`：提前性和迟到信号。
- `PBA-STRUCTURE-*`：结构判断。
- `PBA-INDICATOR-*`：技术指标解释。
- `PBA-TIMEFRAME-*`：多周期推理。
- `PBA-DERIVATIVES-*`：衍生品验证。
- `PBA-RR-*`：风险赔率。
- `PBA-PLAN-*`：交易计划。
- `PBA-REVIEW-*`：复盘进化。

## 7. 验收标准

专业回测 v2 完成后，必须能回答：

1. 网站是否真的提前发现了机会。
2. 网站是否跑赢简单基线。
3. 网站哪些分析规则有效。
4. 网站哪些分析规则误导。
5. 网站为什么漏掉机会。
6. 网站为什么放过垃圾信号。
7. 下一轮必须修什么。

如果不能回答这些问题，不能称为专业回测系统。
