# 专业回测测试方案

本文定义专业回测系统到底要测试什么。它不是量化收益回测，也不是为了做漂亮命中率；它是用历史行情审查川 Market Radar 是否具备核心能力：

```text
快速对全市场覆盖性扫描 -> 发现机会 -> 给出策略 -> 自我提升
```

## 1. 总目标

每一轮正式回测必须回答五个问题：

1. 系统能不能在几百个合约币里提前发现值得关注的山寨币。
2. 系统筛出来的候选是否明显好于随机、涨幅榜、成交额榜、单纯动量榜。
3. 系统是否能判断“方向”和“可交易性”的区别。
4. 系统给出的结构、风控、结构盈亏比、计划和失效条件是否站得住。
5. 系统是否能把漏判、错判、迟到和无计划问题变成下一轮整改方案。

如果一轮回测只给命中率，不解释为什么漏、为什么迟、为什么不能出计划、下一步怎么改，这一轮不算专业回测。

## 2. 必测链路

正式回测必须按真实网站链路逐层测试，不能只测某一个评分函数。

| 链路 | 要测试什么 | 失败表现 |
| --- | --- | --- |
| 数据层 | 历史 K 线、Funding、OI、候选池是否真实可用，是否无未来函数 | 用 0 值、旧缓存、当前值冒充历史数据 |
| 全市场轻扫 | 是否覆盖足够多 USDT 永续，是否只有固定大币 | 只扫 BTC/ETH/SOL 或 Top 币 |
| 候选排序 | 启动前、压缩、早期放量、回踩再确认是否能排进 TopN | 已涨完/跌完的币长期霸榜 |
| 深扫验证 | 衍生品和公开交易所数据是否能解释资金质量 | CoinGlass 不可用时冒充正常 |
| 结构分析 | 前高前低、箱体、支撑压力、突破/假突破、回踩、趋势完整性 | 指标亮就直接给方向 |
| 技术指标 | RSI、MACD、Bollinger、ATR、EMA、VWAP、ADX、OBV/CVD 近似、Fibonacci 是否只做辅助 | 单一指标直接生成结论 |
| 多周期 | 15m/1h/4h/1d 是否冲突，高周期是否能否决低周期 | 低周期信号冲破高周期压力门禁 |
| 提前性 | 信号是启动前、早期、中段、晚到还是事后 | 已经大涨才提示多，已经大跌才提示空 |
| 风险赔率 | 入场、止损、目标、结构盈亏比是否真实，是否至少 3:1 | RR 不足仍出计划 |
| 交易计划 | 只有 TRADE_PLAN_READY 才能出计划，计划必须可执行可失效 | 候选信号被包装成狙击目标 |
| 复盘进化 | 是否能统计命中、失败、超时、最大浮盈、最大回撤、漏判原因 | 只有结果，没有原因和整改方案 |

### 2.1 时间顺序回放协议

正式回测必须按时间顺序回放，不能用完整历史数据直接计算未来涨幅再反推信号。

硬规则：

- 每个扫描点都有明确 `observedAt`。
- 在 `observedAt` 生成候选、信号、证据链、结构判断、技术指标、RR 和交易计划时，只能使用 `observedAt` 之前的数据。
- `observedAt` 之后的 K 线、Funding、OI 或任何结果数据，只能用于该信号生成后的表现评估。
- 未来数据只能评估：是否命中、是否先止损、是否超时、最大浮盈、最大回撤、迟到程度和漏判归因。
- 未来数据不能参与：候选排序、信号生成、方向判断、关键位生成、RR 计算、交易计划生成和证据链生成。
- 正式 `10x10` 审计节点选择必须使用时间桶和 `observedAt` 之前可见特征；不能用未来最大涨跌幅、未来波动率或未来命中结果挑选测试节点。
- 如果某轮先用完整历史数据挑出未来涨幅最大的节点，再回填当时理由，该轮必须判定为无效回测。

## 3. 样本设计

正式轮次默认采用 `10 x 10`：

- 10 个目标山寨币。
- 每个币 10 个不同历史节点。
- 至少 80 个候选币参与同一轮排序竞争。
- TopN 默认 10。

目标币池和候选池必须分开。10 个目标币只是被审计样本，不等于候选排序池。禁止用“10 个币选 Top10”证明扫描有效。

### 3.1 目标币类型

每轮 10 个目标币必须尽量覆盖不同类型：

1. AI / DePIN。
2. DeFi。
3. 交易所生态 / 基建。
4. GameFi / NFT / 消费叙事。
5. 大流动性山寨。
6. L1 / L2。
7. 长尾小币。
8. Meme / 高波动币。
9. 中市值趋势币。
10. 新热点或新上市币。

如果当轮无法覆盖某类，报告必须说明原因，不能用同质币种凑数。

### 3.2 节点类型

每个币 10 个节点必须覆盖：

1. 启动前压缩。
2. 早期放量。
3. 突破边缘。
4. 回踩确认。
5. 趋势加速。
6. 晚到延展。
7. 假突破 / 结构失效。
8. 随机中性节点。
9. 中周期波段节点。
10. 大周期背景节点。

节点不能只挑系统容易赢的时间点。随机中性节点和失败节点必须保留，因为它们能测试系统是否会乱给机会。

### 3.3 验证窗口

节点必须按周期使用不同验证窗口：

- small：默认未来 4 小时。
- medium：默认未来 24 小时。
- large：默认未来 96 小时。

报告必须输出每个节点的验证窗口，不能把短线机会、中线结构和大周期背景混成同一个 24h 结论。

## 4. 对照基线

每轮必须至少对比四条基线：

1. Radar：网站当前完整扫描/分析系统。
2. Momentum：单纯动量或涨跌幅。
3. Volume：单纯成交量。
4. Random：随机选币。

如果 Radar 长期跑不赢 Random，说明系统没有筛选能力。如果 Radar 长期跑不赢 Momentum，说明系统可能只是复杂化但没有增加实战价值。

每条基线至少输出：

- 样本数。
- 命中率。
- 迟到率。
- 平均最大浮盈。
- 平均最大回撤。
- 入选时已波动幅度。
- 入选时成交量倍数。

## 5. 核心指标

每轮必须输出：

- radar TopN 捕获率。
- radar 与 momentum / volume / random 的差距。
- 提前、早期、中段、晚到、事后样本数量。
- 迟到率。
- 无交易计划数量。
- `TRADE_PLAN_READY` 数量。
- 最大浮盈 / 最大回撤。
- 先到 TP、先到 SL、超时数量。
- 未进入 TopN 但事后达到阈值的漏判机会。
- 漏判机会的平均 radar 排名。
- 主要问题编号和根因。

## 6. 问题编号

问题必须按类型归档，方便下一轮整改：

- `PBA-DATA-*`：历史数据、数据质量、未来函数、数据缺失。
- `PBA-SCAN-*`：全市场覆盖、候选池、排序、TopN 捕获。
- `PBA-TIMING-*`：提前性、迟到信号、事后信号。
- `PBA-STRUCTURE-*`：结构判断、关键位、假突破、回踩。
- `PBA-INDICATOR-*`：技术指标误用或缺失。
- `PBA-TIMEFRAME-*`：多周期冲突或门控失效。
- `PBA-DERIVATIVES-*`：OI、Funding、多空拥挤、主动买卖。
- `PBA-RR-*`：结构盈亏比、止损、目标、追涨追空。
- `PBA-PLAN-*`：交易计划生成、成熟度、风控门禁。
- `PBA-REVIEW-*`：复盘归因、样本统计、整改闭环。

## 7. 每轮报告必须回答

每轮报告必须明确回答：

1. 这轮系统有没有提前发现机会。
2. 这轮系统有没有跑赢简单基线。
3. 这轮主要漏掉了什么类型的币。
4. 这轮主要晚在哪里。
5. 这轮为什么没有或很少生成交易计划。
6. 这轮有没有把垃圾机会放进前排。
7. 这轮哪些规则有效。
8. 这轮哪些规则误伤或误导。
9. 下一轮先改哪几个问题。
10. 下一轮验收标准是什么。
11. 对比上一轮有没有提升，哪些指标提升、哪些退步、哪些持平、哪些因为样本或数据口径变化不可比。

### 7.1 上一轮对比

每轮正式报告必须单独输出“对比上一轮”小节。对比不能只写一句“有提升”或“没有提升”，必须按指标逐项说明。

必须对比：

- radar 命中率。
- radar TopN 捕获率。
- 启动前机会捕获率。
- 大周期背景机会捕获率。
- 迟到率。
- 漏判早期命中数。
- `TRADE_PLAN_READY` 数量。
- 高优先级问题数量。
- 计划卡点 Top3 是否变化。
- radar 是否跑赢 random / volume / momentum。

每项必须给出：

- 上一轮数值。
- 本轮数值。
- 变化方向：提升 / 退步 / 持平 / 不可比。
- 简短解释。

如果两轮目标币、候选池、节点、窗口或数据源不同，报告必须标记“样本不可完全横向比较”，但仍要给出趋势判断，不能跳过对比。

## 8. 回测状态分级

每轮结束后必须给系统能力定级：

| 状态 | 含义 |
| --- | --- |
| 不能支撑实战 | 跑不赢随机或问题严重，不能作为实盘参考 |
| 只能观察 | 能发现部分机会，但迟到、漏判、无计划问题明显 |
| 有初步参考价值 | 跑赢随机和部分基线，但仍需连续回测验证 |
| 可进入实战观察 | 多轮稳定跑赢基线，迟到率下降，交易计划质量可复核 |
| 相对成熟 | 多轮、不同市场环境下稳定有效，复盘能持续改善 |

不能因为单轮结果好就宣布成熟。

## 9. 正式工作流

固定流程：

```text
回测
-> 发现问题
-> 问题归因
-> 整改方案
-> 整改
-> 常规验证
-> 用户确认
-> 下一轮回测
```

禁止直接连续跑多轮却不整改。也禁止整改完自动把自己判定为成功。每轮必须先看报告，再决定下一轮要测试什么。

## 10. 通过标准

一轮正式回测只有同时满足以下条件，才算有效：

- 候选池数量显著大于 TopN。
- 至少完成 10 个目标币和 100 个历史节点。
- 数据缺失、partial、unavailable 被明确写出。
- 没有未来函数。
- 能证明扫描点严格按 `observedAt` 顺序回放，信号生成前没有读取未来结果。
- 输出四条基线对比。
- 输出提前性和漏判机会。
- 输出问题编号、根因和整改方案。
- `/review` 能看到最新进度或最新报告。

如果缺少上述任一项，只能叫临时验证，不能叫正式专业回测。
