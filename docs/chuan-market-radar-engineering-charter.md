# 川 Market Radar 长期工程章程

> 本文定义川 Market Radar 如何长期搭建、优化、删除和升级。它不是施工日志，也不是功能清单；它是后续迭代的工程地基。

## 核心目标

川 Market Radar 的核心不是做一个好看的币圈面板。

核心目标是：

**帮助用户在合约市场里更早、更冷静、更有证据地发现异常机会，同时避免追单、误判和情绪化决策。**

所有功能、视觉、角色、AI、数据源、告警和复盘能力，都必须服务这个闭环：

```text
合约市场异常识别 -> 证据分层 -> 风险纪律 -> 复盘进化
```

任何新增、优化或删除，先问：

**它是否增强了异常识别、证据判断、风险纪律或复盘进化？**

如果不能增强这个闭环，即使好看、好玩或技术上有趣，也不能优先做。如果会削弱核心判断，必须延期、拆小或删除。

## 不可偏移边界

- 前端可以更漂亮，但必须让信号更易读，不能为炫技牺牲判断清晰度。
- AI 可以做反证复核、解释归纳和盲区提示，不能成为最终裁决。
- 数据源可以扩展，但必须服务证据完整性，不能为了堆指标污染分析。
- 告警必须提醒高质量异动和系统异常，不能制造追涨杀跌焦虑。
- 段位和纪律反馈必须强化好习惯，不能暗示收益承诺。
- 每日涨跌幅榜只能用于归因、样本库和规则校准，不能成为追高杀跌入口。

## 系统分层

网站长期按 6 层搭建。每个功能必须先归属到一层，再决定怎么实现。

| 层级 | 责任 | 当前主要位置 | 不允许做什么 |
| --- | --- | --- | --- |
| 数据层 | CoinGlass、OHLCV、交易所合约池、缓存、限流、失败降级 | `src/lib/market` | 不直接给交易结论 |
| 分析层 | 异常识别、多周期、技术指标、评分、策略计划、AI 反证边界 | `src/lib/analysis` | 不依赖 UI 状态 |
| 持久化层 | Neon、扫描归档、复盘日记、段位、规则反馈 | `src/lib/persistence`、`src/lib/journal` | 不绕过 repository 合同 |
| 应用编排层 | API、定时扫描、健康状态、迁移、权限、rate limit | `src/app/api`、`src/lib/api` | 不把业务规则散落到路由里 |
| 展示层 | v0 前端 UI 已作为当前展示事实源接入；页面、候选池、图表、复盘、系统状态必须消费 API 契约 | `src/app`、`src/components`、`src/lib/radar-contract.ts` | 不重新发明市场判断逻辑 |
| 体验层 | v0 前端的动效、声音、宠物小人、中文化和移动端体验尽量 1:1 保留 | `src/app/globals.css`、`src/components` | 不越权影响分析结论 |

关键规则：

**体验层不能越权影响分析层，前端不能污染判断逻辑，AI 不能替代规则引擎。**

## 功能准入流程

每次新增、优化或删除功能，先过 5 道门：

1. **核心门**：是否服务“异常识别、证据分层、风险纪律、复盘进化”？
2. **层级门**：它属于数据层、分析层、持久化层、应用编排层、展示层，还是体验层？
3. **边界门**：是否破坏现有模块职责，或让前端、AI 越权？
4. **版本门**：应该现在做，还是放到 V2、V3、V3.x、V4 或更后？
5. **验证门**：需要测试、迁移、本地预览、Vercel 预览、健康检查或模块文档吗？

过不了核心门，不做。过不了边界门，先重新拆设计。

## 删除和降级规则

长期运营的网站必须能干净删除功能。删除前检查：

- 谁依赖它？
- 它依赖谁？
- 删除后是否需要降级状态、空态或兼容层？
- 是否有历史数据要保留、迁移或废弃？
- 删除后是否让核心闭环更清楚？

删除不是失败。不能服务核心闭环、增加误导或拖累维护的功能，应该被删除或降级。

## 文档分级

后续不能把所有变化都写进蓝图。文档分级如下：

| 文档类型 | 记录什么 | 不记录什么 |
| --- | --- | --- |
| 蓝图 `docs/chuan-market-radar-blueprint.md` | 产品定位、长期原则、核心边界、重大路线 | 普通前端小修、文案、局部功能流水账 |
| 工程章程 `docs/chuan-market-radar-engineering-charter.md` | 长期搭建原则、分层、准入、删除、验证规则 | 某个具体按钮怎么写 |
| 模块文档 `docs/modules/*.md` | 单个系统的稳定合同、输入输出、边界 | 临时实验和普通样式改动 |
| ADR `docs/decisions/*.md` | 重大取舍和不可逆决策 | 轻量调整 |
| commit / PR | 日常修改、bugfix、前端优化、测试补充 | 长期原则 |

蓝图是宪法，不是施工日志。普通功能和小优化默认写进 commit 或 PR；只有改变产品原则、模块边界或长期路线时才更新蓝图。

## 每日异动归因复盘

后续可新增“每日异动归因复盘系统”，用于抓取虚拟货币合约涨幅榜和跌幅榜，并与自我进化复盘系统互补。

定位：

**每日涨跌幅榜不是追涨杀跌入口，而是真实市场异动样本库。**

它要回答：

- 今天为什么涨？
- 今天为什么跌？
- 上涨或下跌前 1h、4h、24h、3d 发生了什么？
- 雷达当时是否提前发现？
- 如果发现了，哪些证据有效？
- 如果没有发现，是覆盖率问题、数据源问题、规则问题，还是不可学习事件？
- 哪些前兆应该提升权重？
- 哪些误报应该降低权重？

它与复盘系统的关系：

- 复盘系统看“雷达发现过什么”。
- 涨跌幅榜复盘看“市场真实发生了什么”。
- 两者对比，才能知道系统抓住了什么、漏掉了什么、误判了什么。

必须防止：

- 只研究涨跌幅榜造成幸存者偏差。
- 事后硬解释，把随机事件说成规律。
- 把低流动性插针、新闻突发、一次性拉盘当作可学习规律。
- 把榜单前端做成追涨杀跌入口。

## 验证规则

不同改动需要不同级别验证：

- 文档改动：检查 diff、确认没有与蓝图冲突。
- 分析逻辑改动：必须跑相关单元测试，通常还要跑 `npm run test:market`。
- 数据源、扫描、持久化改动：必须验证失败降级、缓存、schema 或 repository 合同。
- 前端改动：必须跑类型、lint、build；能启动本地服务时，用 Browser/Playwright 检查桌面和移动端。
- 部署相关改动：必须跑 build，并在推送后看 Vercel 预览或生产健康状态。

任何“已完成”都必须有验证证据。不能把骨架说成完整能力，不能把 mock、stale、fallback 说成真实生产数据。

## 后续搭建口令

以后继续开发时，默认按这个顺序：

1. 对照蓝图确认核心目标和边界。
2. 用本章程判断功能归属和准入。
3. 判断是否需要模块文档、ADR 或只需要 commit。
4. 设计最小可验证版本。
5. 实现并跑对应验证。
6. 只把真正影响长期原则、边界或路线的内容写回蓝图。

这套规则的目标是让网站可以长期运营、持续进化，并且越升级越清楚，而不是越改越乱。
